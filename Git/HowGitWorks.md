# Отличие Git от остальных СКВ

Основное отличие Git от остальных СКВ заключается в подходе над работе со своими данными. Большинство СКВ хранят информацию в виде списка изменений в файлах. Эти системы представляют хранимую информацию в виде набора файлов и изменений, сделанных в каждом файле, по времени.

</p>


На этой картинке показан принцып хранения данных основанных на различиях.

Git же работает по другому. Он делает скорее "снимки" нашей файловой системы . Каждый раз когда мы делаем коммит, то есть сохраняем состояние всего проекта в Git, то система запоминает, как выглядел **каждый** файл в этот момент, и сохораняет ссылку на этот снимок. Для эффективности, если файл не был изминен, Git не запоминает этот файл вновь, а только создает ссылку на предыдущую версию этого файла. Git представляет свои данные как, **"поток снимков"**.

Такой подход вносит свои преимущества которые можно будет увидеть при работе с ветками.

## Почти все операции выполняются локально
Огромное отличие Git от ЦСКВ в том что мы имеем всю историю локально. По-этому все операции происходят моментально. Все вычисления происходят локально. В то время как в ЦСКВ все запросы идут на сервер, из-за чего на это тратится время.
Единственное к чему нужен доступ к Интернету это если мы хотим сделать **push** или подтянуть последние изменения.

Git работает с хеш-сумами. Все обьекты сохраняються не по имени, а по хеш сумме. Механизм который использует Git для вычисления хеш суммы называеться SHA-1 хеш.

## Три состояния
У Git есть 3 основных состояния в которых могут находится наши файлы: изменен(modified), индексирован (staged) и зафиксирован(committed):
 - К измененным относятся файлы, которые поменялись, но еще не были зафиксированы
 - Индексированный - это измененный файл в его текущей версии, отмеченный для включения в следующий коммит.
 - Зафиксированный значит, что файл уже сохранен в нашей локальной базе.

По этому у нас есть три основные секции проекта Git: рабочая копия (working directory), область индексирования (staging area) и каталог Git (Git directory).

  <img  src="https://i.imgur.com/RxPulyd.png">

</p>

Рабочая копия является снимком одной версии проекта. Эти файлы извлекаются из сжатой базы данных в каталоге Git и помещаются на диск, для того чтобы их можно было использовать или редактировать.

Область индексирования — это файл, обычно находящийся в каталоге Git, в нём содержится информация о том, что попадёт в следующий коммит. Её техническое название на языке Git — «индекс», но фраза «область индексирования» также работает.

Каталог Git — это то место, где Git хранит метаданные и базу объектов вашего проекта. Это самая важная часть Git и это та часть, которая копируется при клонировании репозитория с другого компьютера.

Базовый подход в работе с Git выглядит так:

Изменяете файлы вашей рабочей копии.

Выборочно добавляете в индекс только те изменения, которые должны попасть в следующий коммит, добавляя тем самым снимки только этих изменений в индекс.

Когда вы делаете коммит, используются файлы из индекса как есть, и этот снимок сохраняется в ваш каталог Git.

Если определённая версия файла есть в каталоге Git, эта версия считается зафиксированной (committed). Если файл был изменён и добавлен в индекс, значит, он индексирован (staged). И если файл был изменён с момента последнего распаковывания из репозитория, но не был добавлен в индекс, он считается изменённым (modified). В главе Основы Git вы узнаете больше об этих состояниях и какую пользу вы можете извлечь из них или как полностью пропустить часть с индексом.
